# RPA底座2.0建设项目（V1.0初版）

版本：V1.0  
项目名称：seebot RPA底座2.0建设项目  
适用范围：社保、公积金、税务网页端、社保费管理客户端、城市网厅、凭证获取、文件处理、增减员、基数申报、费用明细、查审核结果等自动化执行场景  
建设周期：12～16 周  
建设方式：数据驱动、分层治理、先试点、再推广、后平台化

---

## 一、项目背景

seebot 是基于 RPA 技术实现企业级社保、公积金网上申报、缴费、凭证获取等业务自动化的平台。当前 v1.0 已上线，已具备云端任务调度、UKey/CA 远程挂载、执行截图、日志追踪等基础能力，能够支撑现有城市与业务场景的自动化执行。

但是，随着业务城市数量、客户账户数量、缴费任务数量持续增加，v1.0 执行底座暴露出明显的规模化瓶颈。问题已经不只是执行器数量不足，而是执行环境、任务路由、UKey/CA、城市差异、页面变化、配置质量和超时治理共同叠加后的平台能力不足，主要体现在：

1. 忙时执行器资源不足，任务集中进入队列后容易产生等待和积压，固定工作站模式无法根据任务高峰动态扩容；
2. Windows 执行环境长期运行后稳定性下降，容易出现桌面会话异常、进程残留、下载目录残留、登录态污染、驱动冲突等问题；
3. 不同城市、不同 CA、不同 UKey 驱动、不同浏览器版本、不同客户端长期混装，导致环境差异不可控，问题难以复制和回滚；
4. 部分城市对浏览器、社保费管理客户端、CA 控件、UKey 驱动、证书链和安全策略要求严格，依赖人工维护的方式成本高、风险大；
5. seebot v1.0 已具备 UKey/CA 远程挂载能力，但生产数据表明 UKey 挂载、证书识别、资源释放、独占锁和健康检测仍是影响稳定性的关键环节；
6. 客户端类流程高度依赖 Windows 图形界面，Win 图片点击、窗口焦点、分辨率、DPI、弹窗处理等问题会直接导致任务失败；
7. 网页端流程受网厅页面变化、登录策略变化、H5 扫码、短信时间窗口、网站元素定位变化影响明显，单纯增加执行器无法解决脚本和页面回归问题；
8. 配置错误、申报账户不一致、流程配置问题等本应在任务执行前拦截的问题进入 RPA 阶段，造成执行器资源浪费；
9. 失败任务耗时过长，部分任务长时间卡死或等待，缺少分步骤超时、失败快返和城市级熔断机制，容易拖垮整体队列吞吐；
10. 机器维度存在集中失败，说明部分工作站或盒子已成为稳定性风险源，但当前缺少执行器健康评分、自动隔离和重新入池机制。

综合来看，RPA底座2.0 不是单纯扩容项目，也不是单纯补脚本或单点优化 UKey，而是需要把执行环境、执行资源、任务路由、UKey/证书、页面回归、配置前置校验、超时熔断、轻重任务分流和执行器健康治理统一纳入平台化建设。

因此，需要建设 RPA底座2.0，将原有“固定工作站执行模式”升级为“可编排、可扩容、可回滚、可版本化管理、可健康隔离的 RPA 执行平台”。

## 二、项目定位

### 2.1 项目名称

seebot RPA底座2.0建设项目

### 2.2 项目定位

本项目定位为：

> 面向企业级社保、公积金、税务网厅和客户端自动化场景，建设可编排、可扩容、可回滚、可自检、可熔断、可观测的 RPA 执行底座。

### 2.3 不再采用的错误定位

本项目不应被理解为：

1. 单纯购买更多机器；
2. 单纯把 Windows 做成镜像；
3. 单纯把 RPA 迁移到容器；
4. 单纯优化 UKey；
5. 单纯补脚本。

正确定位是：

> 用数据驱动方式，把执行环境、执行资源、任务路由、异常治理、镜像版本、配置校验、页面回归、超时熔断统一纳入 RPA 执行平台治理。

## 三、建设目标

### 3.1 总体目标

建设 RPA底座2.0，将 seebot 1.0 的“固定工作站自动化执行模式”升级为：

> Windows 非持久化 VM 执行池为主、Kubernetes 容器任务池为辅、城市级黄金镜像为标准、执行器健康评分为准入、超时熔断为保障、页面回归和配置前置校验为质量门禁的企业级 RPA 云执行底座。

### 3.2 业务目标

1. 降低缴费、增减员、基数申报、费用明细等核心业务的失败率；
2. 降低 UKey 挂载异常、Win 图片点击失败、设备/盒子/环境异常；
3. 降低长耗时失败任务对队列吞吐的影响；
4. 降低非工作时间短信导致的空跑；
5. 降低配置错误、账户不一致导致的无效执行；
6. 提高高峰期任务吞吐和动态扩容能力；
7. 支撑更多城市、更多账号、更多申报系统接入。

### 3.3 技术目标

1. 建设 Windows RPA 非持久化 VM 执行池；
2. 建设城市级黄金镜像体系；
3. 建设 Kubernetes 轻任务容器池；
4. 建设执行资源路由器；
5. 建设执行器健康评分与自动隔离机制；
6. 建设执行前环境自检和 UKey/证书自检；
7. 建设分步骤超时、失败快返和熔断机制；
8. 建设城市级页面回归和脚本版本管理机制；
9. 建设配置前置校验机制；
10. 建设结构化异常码和步骤码体系。

## 五、核心建设内容

### 5.1 Windows RPA 非持久化 VM 执行池

#### 建设原因

三个月数据中，`UKey挂载异常`、`Win图片点击失败`、`设备/盒子/环境异常`、`证书识别/读取失败` 等环境相关问题合计占比较高。固定工作站长期运行后，会形成环境污染、驱动冲突、桌面会话异常、机器个体差异。

#### 建设目标

将固定工作站升级为可申请、可分配、可自检、可回滚、可销毁、可重新入池的非持久化执行器。

#### 核心能力

1. VM 热池管理；
2. VM 冷池管理；
3. 按城市镜像创建执行器；
4. Worker Agent 自动注册；
5. 执行前环境自检；
6. UKey/证书挂载检测；
7. 任务完成后回滚或销毁；
8. 执行器健康评分；
9. 高风险机器自动隔离；
10. 任务失败快返和资源释放。


#### 优先支持城市

建议首批试点城市：

| 城市 | 主要问题 | 建设重点 |
|---|---|---|
| 呼和浩特 | UKey挂载异常、Win图片点击失败 | UKey 自检、客户端图像操作、城市镜像 |
| 成都 | Win图片点击失败 | Windows 桌面标准化、DPI/分辨率/窗口焦点治理 |
| 天津 | UKey挂载异常、网站元素操作异常 | UKey 自检、页面回归、脚本版本治理 |
| 苏州 | UKey挂载异常 | UKey/证书/驱动镜像隔离 |
| 深圳 | Win图片点击失败、空跑 | 桌面标准化、短信时间窗口调度 |
| 广州 | 空跑高、登录与短信策略 | 调度时间窗口、配置前置校验 |

不建议一开始全量城市切换，应选择 3～5 个城市做试点。

---

### 5.2 城市级黄金镜像体系

#### 建设原因

城市差异是 RPA 稳定性的核心变量。不同城市存在不同网厅、CA、UKey、浏览器、客户端、征缴中心、短信规则和下载规则。继续混装会导致环境污染和不可复制。

#### 镜像分层

```text
基础 OS 镜像
    |
    v
RPA 通用镜像
    |
    v
城市专用镜像
```

#### 基础 OS 镜像包含

1. Windows 系统；
2. 系统补丁；
3. VMware Tools；
4. 基础安全策略；
5. 远程管理配置；
6. 字体、输入法、分辨率、DPI；
7. 禁用不必要的自动升级；
8. 固定下载目录和临时目录策略。

#### RPA 通用镜像包含

1. RPA Worker Agent；
2. 日志采集组件；
3. 截图组件；
4. OCR 基础依赖；
5. 文件上传工具；
6. 健康检查脚本；
7. 超时熔断组件；
8. Agent 注册脚本。

#### 城市专用镜像包含

1. 城市 CA 驱动；
2. 城市 UKey 驱动；
3. 社保费管理客户端；
4. 公积金客户端；
5. 浏览器版本和策略；
6. IE Mode 配置；
7. 可信站点；
8. 证书链；
9. 城市页面回归脚本；
10. 城市登录前置检查脚本。

#### 镜像命名规范

```text
rpa-env-{city}-{business}-{system}-{version}
```

示例：

```text
rpa-env-hohhot-social-client-202606
rpa-env-chengdu-social-client-202606
rpa-env-tianjin-social-web-202606
rpa-env-suzhou-social-ukey-202606
rpa-env-shenzhen-social-taxweb-202606
```

### 5.3 Kubernetes 容器任务池

#### 建设原因

Windows 执行器是稀缺重资源，不应被文件处理、OCR、PDF、Excel、账单生成、凭证打包等轻任务占用。

#### 适合迁移的任务

1. Excel 文件解析；
2. PDF 文件解析；
3. PDF 转图片；
4. OCR 后处理；
5. 截图金额校验；
6. 明细附件字段识别；
7. 汇总账单生成；
8. 凭证文件整理；
9. ZIP 打包；
10. 无 UKey 纯网页下载；
11. 报盘文件有效性校验；
12. 接口同步任务。

#### 不适合迁移的任务

1. 需要 UKey 登录；
2. 需要 CA 控件；
3. 需要社保费管理客户端；
4. 需要 Windows 图片点击；
5. 需要 IE 控件或本地安全控件；
6. 核心缴费、核定、客户端操作流程。

### 5.4 执行器健康评分与自动隔离

#### 建设原因

Top 执行器持续产生大量失败，说明 seebot 1.0 缺少机器健康准入机制。

#### 健康评分指标

| 指标 | 说明 |
|---|---|
| 最近 24 小时失败率 | 判断短期稳定性 |
| 最近 7 天失败率 | 判断长期稳定性 |
| UKey 异常次数 | 判断 UKey 挂载稳定性 |
| Win 图片点击失败次数 | 判断桌面会话稳定性 |
| 设备/盒子/环境异常次数 | 判断机器环境健康 |
| 平均任务耗时 | 判断是否卡顿 |
| 超时任务数 | 判断是否容易占用资源 |
| Agent 心跳稳定性 | 判断执行器在线质量 |
| 镜像版本匹配度 | 判断环境是否合规 |
| 最近一次自检结果 | 判断是否允许入池 |

#### 自动隔离规则

1. 连续失败 3 次自动进入观察状态；
2. 1 小时内环境类失败超过阈值，自动下线；
3. UKey 异常后强制释放 UKey 锁；
4. Win 图片点击失败连续出现，执行器回滚或销毁；
5. Agent 心跳丢失，禁止分配新任务；
6. 被隔离机器必须通过环境自检和冒烟任务后才能重新入池。

---

### 5.5 超时熔断与失败快返

#### 建设原因

5 月失败任务 P90 已达到 **735.8 分钟**，这会直接拖垮执行池。长耗时失败必须优先治理。

#### 分步骤超时策略

| 步骤类型 | 建议超时 |
|---|---:|
| 打开网站/客户端 | 3～5 分钟 |
| 登录 | 5～10 分钟 |
| 短信等待 | 按城市配置，不允许无限等待 |
| 页面元素等待 | 1～3 分钟 |
| 文件下载 | 5～10 分钟 |
| 报盘生成 | 10～20 分钟 |
| 核定提交 | 10～20 分钟 |
| 缴费提交 | 10～30 分钟 |
| 凭证获取 | 按城市异步策略配置 |

#### 熔断规则

1. 单步骤超时后立即截图、采集日志、返回结构化异常；
2. 同一任务最多重试 N 次；
3. 同一账号同一流程连续失败后暂停自动重试；
4. 同一城市同一系统批量失败时触发城市级熔断；
5. 同一镜像版本批量失败时触发镜像阻断；
6. 同一页面元素批量失败时触发脚本回归告警。

---

### 5.6 页面回归与脚本版本治理

#### 建设原因

三个月 `网站元素操作异常` 达到 **32,716**，是最高频失败原因之一。该类问题不能通过 VM 扩容解决。

#### 建设内容

1. 对关键城市、关键系统建立页面指纹；
2. 对登录、在册、汇缴、报盘、核定、缴费等页面建立元素定位基线；
3. 每日执行城市级冒烟任务；
4. 页面元素变化时提前告警；
5. 脚本适配完成后绑定城市镜像版本发布；
6. 对高频失败步骤建立自动回归用例。

#### 首批回归步骤

1. 登录；
2. 在册主流程；
3. 社保客户端登录模版；
4. 社保客户端-缴费核定子流程-新版；
5. 增员基数申报核验子流程；
6. 访问汇缴业务；
7. 养老登录；
8. 医疗登录；
9. 登录公积金申报网站；
10. 生成报盘文件。

---

### 5.7 配置前置校验

#### 建设原因

三个月 `配置错误` 失败 **24,885**，`申报账户不一致` 失败 **4,365**，`流程配置问题` 失败 **3,720**。这些问题不应进入 RPA 执行阶段。

#### 前置校验项

1. 申报账户是否存在；
2. 账号与城市是否匹配；
3. 账号与申报系统是否匹配；
4. UKey 与账号是否绑定；
5. 缴费方式是否配置；
6. 银行选项是否配置；
7. 网站缴款方式是否匹配城市枚举；
8. 是否在可操作时间窗口；
9. 是否需要短信且当前是否允许短信；
10. 是否存在未完成同类任务；
11. 是否存在账号锁；
12. 是否配置城市环境画像；
13. 是否存在可用稳定镜像。

#### 处理原则

配置错误不进入执行器。前置校验不通过的任务，应直接失败快返或进入人工配置修复队列。

## 六、实施路线

### 阶段 0：异常基线与治理口径统一

周期：第 1 周

主要工作：

1. 固化 2026-03 ～ 2026-05 三个月异常基线；
2. 统一失败原因字典；
3. 统一失败步骤字典；
4. 识别高风险城市、高风险机器、高风险业务；
5. 明确底座治理、脚本治理、配置治理的边界。

交付物：

1. 三个月异常基线报告；
2. 异常码字典；
3. 步骤码字典；

## 九、主要风险与应对措施

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 误把本项目做成单纯扩容 | 失败率不会明显下降 | 先做健康评分、熔断、配置校验，再扩容 |
| VM 启动慢 | 任务等待时间增加 | 热池+冷池；高峰期提前预热 |
| 城市镜像数量爆炸 | 维护复杂 | 镜像粒度控制为城市+业务+系统，不做到客户级 |
| UKey 挂载仍不稳定 | 核心流程失败 | 增加挂载自检、证书枚举、失败释放、健康评分 |
| 非持久化环境丢失状态 | 部分流程不兼容 | 必要配置写入镜像或启动脚本，不依赖运行后状态 |
| 页面元素变更频繁 | 网页端仍高失败 | 页面回归、脚本版本治理、城市冒烟机制 |
| 配置错误继续进入执行器 | 浪费资源 | 配置前置校验，不通过不下发 RPA |
| 容器误承接重任务 | 核心流程失败 | 明确容器准入规则，UKey/CA/客户端任务不得迁移 |
| 长耗时任务继续拖垮队列 | 执行器被占满 | 分步骤超时、熔断、失败快返 |
| 异常归因不准 | 治理方向偏差 | 统一异常码和步骤码，减少未填写 |
| 数据合规风险 | 个人信息泄露 | 临时文件清理、日志脱敏、短期访问 token |

## 十一、建设建议

RPA底座2.0 应按以下顺序推进：

1. **先止血**：上线健康评分、自动隔离、超时熔断、配置前置校验，减少病机和长耗时任务继续消耗资源；
2. **再隔离环境**：建设 Windows 非持久化 VM 执行池和城市级黄金镜像，解决环境污染、驱动冲突、机器差异；
3. **同步做质量门禁**：建设页面回归、脚本版本治理、异常码和步骤码，降低网页端批量失败；
4. **再做资源分流**：建设 Kubernetes 容器池，把文件/OCR/PDF/Excel 等轻任务迁移出去；
5. **最后规模化推广**：按高失败城市和高价值业务逐步推广，不建议一次性全量切换。

一句话总结：

> RPA底座2.0 的核心不是“多跑几台机器”，而是把执行环境、执行器健康、UKey/证书、页面变化、配置质量、超时熔断和任务分流统一纳入平台化治理。

只有这样，seebot 才能从 v1.0 的“固定工作站自动化”升级为真正可规模化交付的企业级 RPA 云执行底座。
